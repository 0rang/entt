14 `EnTT` is a header-only, tiny and easy to use library for game programming and
15 much more written in **modern C++**, mainly known for its innovative
16 **entity-component-system (ECS)** model.<br/>
17 [Among others](https://github.com/skypjack/entt/wiki/EnTT-in-Action), it's used
18 in [**Minecraft**](https://minecraft.net/en-us/attribution/) by Mojang, the
19 [**ArcGIS Runtime SDKs**](https://developers.arcgis.com/arcgis-runtime/) by Esri
20 and [**The Forge**](https://github.com/ConfettiFX/The-Forge) by Confetti. Read
21 on to find out what it can offer you.

58 # Introduction
59 
60 The entity-component-system (also known as _ECS_) is an architectural pattern
61 used mostly in game development. For further details:
62 
63 * [Entity Systems Wiki](http://entity-systems.wikidot.com/)
64 * [Evolve Your Hierarchy](http://cowboyprogramming.com/2007/01/05/evolve-your-heirachy/)
65 * [ECS on Wikipedia](https://en.wikipedia.org/wiki/Entity%E2%80%93component%E2%80%93system)
66 
67 This project started off as a pure entity-component system. Over time the
68 codebase has grown as more and more classes and functionalities were added.<br/>
69 Here is a brief, yet incomplete list of what it offers today:
70 
71 * Statically generated integer **identifiers for types** (assigned either at
72  compile-time or at runtime).
73 * A `constexpr` utility for **human readable resource identifiers**.
74 * A minimal **configuration system** built using the monostate pattern.
75 * An incredibly fast **entity-component system** based on sparse sets, with its
76  own _pay for what you use_ policy to adjust performance and memory usage
77  according to the users' requirements.
78 * Views and groups to iterate entities and components and allow different access
79  patterns, from **perfect SoA** to fully random.
80 * A lot of **facilities** built on top of the entity-component system to support
81  the users and avoid reinventing the wheel (dependencies, snapshot, actor class
82  for those who aren't confident with the architecture and so on).
83 * The smallest and most basic implementation of a **service locator** ever seen.
84 * A built-in, non-intrusive and macro-free **runtime reflection system**.
85 * A **cooperative scheduler** for processes of any type.
86 * All that is needed for **resource management** (cache, loaders, handles).
87 * **Delegates**, **signal handlers** (with built-in support for collectors) and
88  a tiny **event dispatcher** for immediate and delayed events to integrate in
89  loops.
90 * A general purpose **event emitter** as a CRTP idiom based class template.
91 * And **much more**! Check out the
92  [**wiki**](https://github.com/skypjack/entt/wiki).
93 
94 Consider this list a work in progress as well as the project. The whole API is
95 fully documented in-code for those who are brave enough to read it.
96 
97 Currently, `EnTT` is tested on Linux, Microsoft Windows and OSX. It has proven
98 to work also on both Android and iOS.<br/>
99 Most likely it won't be problematic on other systems as well, but it hasn't been
100 sufficiently tested so far.

102 ## Code Example
103 
104 ```cpp
105 #include <entt/entt.hpp>
106 #include <cstdint>
107 
108 struct position {
109  float x;
110  float y;
111 };
112 
113 struct velocity {
114  float dx;
115  float dy;
116 };
117 
118 void update(entt::registry &registry) {
119  auto view = registry.view<position, velocity>();
120 
121  for(auto entity: view) {
122  // gets only the components that are going to be used ...
123 
124  auto &vel = view.get<velocity>(entity);
125 
126  vel.dx = 0.;
127  vel.dy = 0.;
128 
129  // ...
130  }
131 }
132 
133 void update(std::uint64_t dt, entt::registry &registry) {
134  registry.view<position, velocity>().each([dt](auto &pos, auto &vel) {
135  // gets all the components of the view at once ...
136 
137  pos.x += vel.dx * dt;
138  pos.y += vel.dy * dt;
139 
140  // ...
141  });
142 }
143 
144 int main() {
145  entt::registry registry;
146  std::uint64_t dt = 16;
147 
148  for(auto i = 0; i < 10; ++i) {
149  auto entity = registry.create();
150  registry.assign<position>(entity, i * 1.f, i * 1.f);
151  if(i % 2 == 0) { registry.assign<velocity>(entity, i * .1f, i * .1f); }
152  }
153 
154  update(dt, registry);
155  update(registry);
156 
157  // ...
158 }
159 ```

161 ## Motivation
162 
163 I started developing `EnTT` for the _wrong_ reason: my goal was to design an
164 entity-component system to beat another well known open source solution both in
165 terms of performance and possibly memory usage.<br/>
166 In the end, I did it, but it wasn't very satisfying. Actually it wasn't
167 satisfying at all. The fastest and nothing more, fairly little indeed. When I
168 realized it, I tried hard to keep intact the great performance of `EnTT` and to
169 add all the features I wanted to see in *my own library* at the same time.
170 
171 Nowadays, `EnTT` is finally what I was looking for: still faster than its
172 _competitors_, lower memory usage in the average case, a really good API and an
173 amazing set of features. And even more, of course.
174 
175 ## Performance
176 
177 As it stands right now, `EnTT` is just fast enough for my requirements when
178 compared to my first choice (it was already amazingly fast actually).<br/>
179 Below is a comparison between the two (both of them compiled with GCC 7.3.0 on a
180 Dell XPS 13 from mid 2014):
181 
182 | Benchmark | EntityX (compile-time) | EnTT |
183 |-----------|-------------|-------------|
184 | Create 1M entities | 0.0147s | **0.0046s** |
185 | Destroy 1M entities | 0.0053s | **0.0045s** |
186 | 1M entities, one component | 0.0012s | **1.9e-07s** |
187 | 1M entities, two components | 0.0012s | **3.8e-07s** |
188 | 1M entities, two components<br/>Half of the entities have all the components | 0.0009s | **3.8e-07s** |
189 | 1M entities, two components<br/>One of the entities has all the components | 0.0008s | **1.0e-06s** |
190 | 1M entities, five components | 0.0010s | **7.0e-07s** |
191 | 1M entities, ten components | 0.0011s | **1.2e-06s** |
192 | 1M entities, ten components<br/>Half of the entities have all the components | 0.0010s | **1.2e-06s** |
193 | 1M entities, ten components<br/>One of the entities has all the components | 0.0008s | **1.2e-06s** |
194 | Sort 150k entities, one component<br/>Arrays are in reverse order | - | **0.0036s** |
195 | Sort 150k entities, enforce permutation<br/>Arrays are in reverse order | - | **0.0005s** |
196 | Sort 150k entities, one component<br/>Arrays are almost sorted, std::sort | - | **0.0035s** |
197 | Sort 150k entities, one component<br/>Arrays are almost sorted, insertion sort | - | **0.0007s** |
198 
199 Note: The default version of `EntityX` (`master` branch) wasn't added to the
200 comparison because it's already much slower than its compile-time counterpart.
201 
202 Pretty interesting results, aren't them? In fact, these benchmarks are the ones
203 used by `EntityX` to show _how fast it is_. To be honest, they aren't so good
204 and these results shouldn't be taken too seriously (indeed they are completely
205 unrealistic).<br/>
206 The proposed entity-component system is incredibly fast to iterate entities,
207 this is a fact. The compiler can make a lot of optimizations because of how
208 `EnTT` works, even more when components aren't used at all. This is exactly the
209 case for these benchmarks. On the other hand, if we consider real world cases,
210 `EnTT` is somewhere between a bit and much faster than the other solutions
211 around when users also access the components and not just the entities, although
212 it isn't as fast as reported by these benchmarks.<br/>
213 This is why they are completely wrong and cannot be used to evaluate any of the
214 entity-component-system libraries out there.
215 
216 The choice to use `EnTT` should be based on its carefully designed API, its
217 set of features and the general performance, not because some single benchmark
218 shows it to be the fastest tool available.
219 
220 In the future I'll likely try to get even better performance while still adding
221 new features, mainly for fun.<br/>
222 If you want to contribute and/or have suggestions, feel free to make a PR or
223 open an issue to discuss your idea.

227 ## Requirements
228 
229 To be able to use `EnTT`, users must provide a full-featured compiler that
230 supports at least C++17.<br/>
231 The requirements below are mandatory to compile the tests and to extract the
232 documentation:
233 
234 * CMake version 3.2 or later.
235 * Doxygen version 1.8 or later.
236 
237 If you are looking for a C++14 version of `EnTT`, check out the git tag `cpp14`.
238 
239 ## Library
240 
241 `EnTT` is a header-only library. This means that including the `entt.hpp` header
242 is enough to include the library as a whole and use it. For those who are
243 interested only in the entity-component system, consider to include the sole
244 `entity/registry.hpp` header instead.<br/>
245 It's a matter of adding the following line to the top of a file:
246 
247 ```cpp
248 #include <entt/entt.hpp>
249 ```
250 
251 Use the line below to include only the entity-component system instead:
252 
253 ```cpp
254 #include <entt/entity/registry.hpp>
255 ```
256 
257 Then pass the proper `-I` argument to the compiler to add the `src` directory to
258 the include paths.

286 ## Tests
287 
288 To compile and run the tests, `EnTT` requires *googletest*.<br/>
289 `cmake` will download and compile the library before compiling anything else.
290 In order to build the tests, set the CMake option `BUILD_TESTING` to `ON`.
291 
292 To build the most basic set of tests:
293 
294 * `$ cd build`
295 * `$ cmake -DBUILD_TESTING=ON ..`
296 * `$ make`
297 * `$ make test`
298 
299 Note that benchmarks are not part of this set.
