19 # Introduction
20 
21 Reflection (or rather, its lack) is a trending topic in the C++ world and, in
22 the specific case of `EnTT`, a tool that can unlock a lot of other features. I
23 looked for a third-party library that met my needs on the subject, but I always
24 came across some details that I didn't like: macros, being intrusive, too many
25 allocations. In one word: unsatisfactory.<br/>
26 I finally decided to write a built-in, non-intrusive and macro-free runtime
27 reflection system for `EnTT`. Maybe I didn't do better than others or maybe yes,
28 time will tell me, but at least I can model this tool around the library to
29 which it belongs and not vice versa.
30 
31 # Reflection in a nutshell
32 
33 Reflection always starts from real types (users cannot reflect imaginary types
34 and it would not make much sense, we wouldn't be talking about reflection
35 anymore).<br/>
36 To _reflect_ a type, the library provides the `reflect` function:
37 
38 ```cpp
39 auto factory = entt::reflect<my_type>("reflected_type");
40 ```
41 
42 It accepts the type to reflect as a template parameter and an optional name as
43 an argument. Names are important because users can retrieve meta types at
44 runtime by searching for them by name. However, there are cases in which users
45 can be interested in adding features to a reflected type so that the reflection
46 system can use it correctly under the hood, but they don't want to allow
47 searching the type by name.<br/>
48 In both cases, the returned value is a factory object to use to continue
49 building the meta type.
50 
51 A factory is such that all its member functions returns the factory itself.
52 It can be used to extend the reflected type and add the following:
53 
54 * _Constructors_. Actual constructors can be assigned to a reflected type by
55  specifying their list of arguments. Free functions (namely, factories) can be
56  used as well, as long as the return type is the expected one. From a client's
57  point of view, nothing changes if a constructor is a free function or an
58  actual constructor.<br/>
59  Use the `ctor` member function for this purpose:
60 
61  ```cpp
62  entt::reflect<my_type>("reflected").ctor<int, char>().ctor<&factory>();
63  ```
64 
65 * _Destructors_. Free functions can be set as destructors of reflected types.
66  The purpose is to give users the ability to free up resources that require
67  special treatment before an object is actually destroyed.<br/>
68  Use the `dtor` member function for this purpose:
69 
70  ```cpp
71  entt::reflect<my_type>("reflected").dtor<&destroy>();
72  ```
73 
74 * _Data members_. Both real data members of the underlying type and static and
75  global variables, as well as constants of any kind, can be attached to a meta
76  type. From a client's point of view, all the variables associated with the
77  reflected type will appear as if they were part of the type itself.<br/>
78  Use the `data` member function for this purpose:
79 
80  ```cpp
81  entt::reflect<my_type>("reflected")
82  .data<&my_type::static_variable>("static")
83  .data<&my_type::data_member>("member")
84  .data<&global_variable>("global");
85  ```
86 
87  This function requires as an argument the name to give to the meta data once
88  created. Users can then access meta data at runtime by searching for them by
89  name.<br/>
90  Data members can be set also by means of a couple of functions, namely a
91  setter and a getter. Setters and getters can be either free functions, member
92  functions or mixed ones, as long as they respect the required signatures.<br/>
93  Refer to the inline documentation for all the details.
94 
95 * _Member functions_. Both real member functions of the underlying type and free
96  functions can be attached to a meta type. From a client's point of view, all
97  the functions associated with the reflected type will appear as if they were
98  part of the type itself.<br/>
99  Use the `func` member function for this purpose:
100 
101  ```cpp
102  entt::reflect<my_type>("reflected")
103  .func<&my_type::static_function>("static")
104  .func<&my_type::member_function>("member")
105  .func<&free_function>("free");
106  ```
107 
108  This function requires as an argument the name to give to the meta function
109  once created. Users can then access meta functions at runtime by searching for
110  them by name.
111 
112 * _Base classes_. A base class is such that the underlying type is actually
113  derived from it. In this case, the reflection system tracks the relationship
114  and allows for implicit casts at runtime when required.<br/>
115  Use the `base` member function for this purpose:
116 
117  ```cpp
118  entt::reflect<derived_type>("derived").base<base_type>();
119  ```
120 
121  From now on, wherever a `base_type` is required, an instance of `derived_type`
122  will also be accepted.
123 
124 * _Conversion functions_. Actual types can be converted, this is a fact. Just
125  think of the relationship between a `double` and an `int` to see it. Similar
126  to bases, conversion functions allow users to define conversions that will be
127  implicitly performed by the reflection system when required.<br/>
128  Use the `conv` member function for this purpose:
129 
130  ```cpp
131  entt::reflect<double>().conv<int>();
132  ```
133 
134 That's all, everything users need to create meta types and enjoy the reflection
135 system. At first glance it may not seem that much, but users usually learn to
136 appreciate it over time.<br/>
137 Also, do not forget what these few lines hide under the hood: a built-in,
138 non-intrusive and macro-free system for reflection in C++. Features that are
139 definitely worth the price, at least for me.

141 # Any as in any type
142 
143 The reflection system comes with its own meta any type. It may seem redundant
144 since C++17 introduced `std::any`, but it is not.<br/>
145 In fact, the _type_ returned by an `std::any` is a const reference to an
146 `std::type_info`, an implementation defined class that's not something everyone
147 wants to see in a software. Furthermore, the class `std::type_info` suffers from
148 some design flaws and there is even no way to _convert_ an `std::type_info` into
149 a meta type, thus linking the two worlds.
150 
151 A meta any object provides an API similar to that of its most famous counterpart
152 and serves the same purpose of being an opaque container for any type of
153 value.<br/>
154 It minimizes the allocations required, which are almost absent thanks to _SBO_
155 techniques. In fact, unless users deal with _fat types_ and create instances of
156 them though the reflection system, allocations are at zero.
157 
158 A meta any object can be created by any other object or as an empty container
159 to initialize later:
160 
161 ```cpp
162 // a meta any object that contains an int
163 entt::meta_any any{0};
164 
165 // an empty meta any object
166 entt::meta_any empty{};
167 ```
168 
169 It can be constructed or assigned by copy and move and it takes the burden of
170 destroying the contained object when required.<br/>
171 A meta any object has a `type` member function that returns the meta type of the
172 contained value, if any. The member functions `can_cast` and `can_convert` are
173 used to know if the underlying object has a given type as a base or if it can be
174 converted implicitly to it. Similarly, `cast` and `convert` do what they promise
175 and return the expected value.

177 # Enjoy the runtime
178 
179 Once the web of reflected types has been constructed, it's a matter of using it
180 at runtime where required.<br/>
181 All this has the great merit that, unlike the vast majority of the things
182 present in this library and closely linked to the compile-time, the reflection
183 system stands in fact as a non-intrusive tool for the runtime.
184 
185 To search for a reflected type there are two options: by type or by name. In
186 both cases, the search can be done by means of the `resolve` function:
187 
188 ```cpp
189 // search for a reflected type by type
190 auto by_type = entt::resolve<my_type>();
191 
192 // search for a reflected type by name
193 auto by_name = entt::resolve("reflected_type");
194 ```
195 
196 There exits also a third overload of the `resolve` function to use to iterate
197 all the reflected types at once:
198 
199 ```cpp
200 resolve([](auto type) {
201  // ...
202 });
203 ```
204 
205 In all cases, the returned value is an instance of `meta_type`. This type of
206 objects offer an API to know the _runtime name_ of the type, to iterate all the
207 meta objects associated with them and even to build or destroy instances of the
208 underlying type.<br/>
209 Refer to the inline documentation for all the details.
210 
211 The meta objects that compose a meta type are accessed in the following ways:
212 
213 * _Meta constructors_. They are accessed by types of arguments:
214 
215  ```cpp
216  auto ctor = entt::resolve<my_type>().ctor<int, char>();
217  ```
218 
219  The returned type is `meta_ctor` and may be invalid if there is no constructor
220  that accepts the supplied arguments or at least some types from which they are
221  derived or to which they can be converted.<br/>
222  A meta constructor offers an API to know the number of arguments, the expected
223  meta types and to invoke it, therefore to construct a new instance of the
224  underlying type.
225 
226 * _Meta destructor_. It's returned by a dedicated function:
227 
228  ```cpp
229  auto dtor = entt::resolve<my_type>().dtor();
230  ```
231 
232  The returned type is `meta_dtor` and may be invalid if there is no custom
233  destructor set for the given meta type.<br/>
234  All what a meta destructor has to offer is a way to invoke it on a given
235  instance. Be aware that the result may not be what is expected.
236 
237 * _Meta data_. They are accessed by name:
238 
239  ```cpp
240  auto data = entt::resolve<my_type>().data("member");
241  ```
242 
243  The returned type is `meta_data` and may be invalid if there is no meta data
244  object associated with the given name.<br/>
245  A meta data object offers an API to query the underlying type (ie to know if
246  it's a const or a static one), to get the meta type of the variable and to set
247  or get the contained value.
248 
249 * _Meta functions_. They are accessed by name:
250 
251  ```cpp
252  auto func = entt::resolve<my_type>().func("member");
253  ```
254 
255  The returned type is `meta_func` and may be invalid if there is no meta
256  function object associated with the given name.<br/>
257  A meta function object offers an API to query the underlying type (ie to know
258  if it's a const or a static function), to know the number of arguments, the
259  meta return type and the meta types of the parameters. In addition, a meta
260  function object can be used to invoke the underlying function and then get the
261  return value in the form of meta any object.
262 
263 
264 * _Meta bases_. They are accessed through the name of the base types:
265 
266  ```cpp
267  auto base = entt::resolve<derived_type>().base("base");
268  ```
269 
270  The returned type is `meta_base` and may be invalid if there is no meta base
271  object associated with the given name.<br/>
272  Meta bases aren't meant to be used directly, even though they are freely
273  accessible. They expose only a few methods to use to know the meta type of the
274  base class and to convert a raw pointer between types.
275 
276 * _Meta conversion functions_. They are accessed by type:
277 
278  ```cpp
279  auto conv = entt::resolve<double>().conv<int>();
280  ```
281 
282  The returned type is `meta_conv` and may be invalid if there is no meta
283  conversion function associated with the given type.<br/>
284  The meta conversion functions are as thin as the meta bases and with a very
285  similar interface. The sole difference is that they return a newly created
286  instance wrapped in a meta any object when they convert between different
287  types.
288 
289 All the objects thus obtained as well as the meta types can be explicitly
290 converted to a boolean value to check if they are valid:
291 
292 ```cpp
293 auto func = entt::resolve<my_type>().func("member");
294 
295 if(func) {
296  // ...
297 }
298 ```
299 
300 Furthermore, all meta objects with the exception of meta destructors can be
301 iterated through an overload that accepts a callback through which to return
302 them. As an example:
303 
304 ```cpp
305 entt::resolve<my_type>().data([](auto data) {
306  // ...
307 });
308 ```
309 
310 A meta type can also be used to `construct` or `destroy` actual instances of the
311 underlying type.<br/>
312 In particular, the `construct` member function accepts a variable number of
313 arguments and searches for a match. It returns a `meta_any` object that may or
314 may not be initialized, depending on whether a suitable constructor has been
315 found or not. On the other side, the `destroy` member function accepts instances
316 of `meta_any` as well as actual objects by reference and invokes the registered
317 destructor if any or a default one.<br/>
318 Be aware that the result of a call to `destroy` may not be what is expected.
319 
320 Meta types and meta objects in general contain much more than what is said: a
321 plethora of functions in addition to those listed whose purposes and uses go
322 unfortunately beyond the scope of this document.<br/>
323 I invite anyone interested in the subject to look at the code, experiment and
324 read the official documentation to get the best out of this powerful tool.

326 # Named constants and enums
327 
328 A special mention should be made for constant values and enums. It wouldn't be
329 necessary, but it will help distracted readers.
330 
331 As mentioned, the `data` member function can be used to reflect constants of any
332 type among the other things.<br/>
333 This allows users to create meta types for enums that will work exactly like any
334 other meta type built from a class. Similarly, arithmetic types can be enriched
335 with constants of special meaning where required.<br/>
336 Personally, I find it very useful not to export what is the difference between
337 enums and classes in C++ directly in the space of the reflected types.
338 
339 All the values thus exported will appear to users as if they were constant data
340 members of the reflected types.
341 
342 Exporting constant values or elements from an enum is as simple as ever:
343 
344 ```cpp
345 entt::reflect<my_enum>()
346  .data<my_enum::a_value>("a_value")
347  .data<my_enum::another_value>("another_value");
348 
349 entt::reflect<int>().data<2048>("max_int");
350 ```
351 
352 It goes without saying that accessing them is trivial as well. It's a matter of
353 doing the following, as with any other data member of a meta type:
354 
355 ```cpp
356 auto value = entt::resolve<my_enum>().data("a_value").get({}).cast<my_enum>();
357 auto max = entt::resolve<int>().data("max_int").get({}).cast<int>();
358 ```
359 
360 As a side note, remember that all this happens behind the scenes without any
361 allocation because of the small object optimization performed by the meta any
362 class.

364 # Properties and meta objects
365 
366 Sometimes (ie when it comes to creating an editor) it might be useful to be able
367 to attach properties to the meta objects created. Fortunately, this is possible
368 for most of them.<br/>
369 To attach a property to a meta object, no matter what as long as it supports
370 properties, it is sufficient to provide an object at the time of construction
371 such that `std::get<0>` and `std::get<1>` are valid for it. In other terms, the
372 properties are nothing more than key/value pairs users can put in an
373 `std::pair`. As an example:
374 
375 ```cpp
376 entt::reflect<my_type>("reflected", std::make_pair("tooltip"_hs, "message"));
377 ```
378 
379 The meta objects that support properties offer then a couple of member functions
380 named `prop` to iterate them at once and to search a specific property by key:
381 
382 ```cpp
383 // iterate all the properties of a meta type
384 entt::resolve<my_type>().prop([](auto prop) {
385  // ...
386 });
387 
388 // search for a given property by name
389 auto prop = entt::resolve<my_type>().prop("tooltip"_hs);
390 ```
391 
392 Meta properties are objects having a fairly poor interface, all in all. They
393 only provide the `key` and the `value` member functions to be used to retrieve
394 the key and the value contained in the form of meta any objects, respectively.

396 # Unregister types
397 
398 A type registered with the reflection system can also be unregistered. This
399 means unregistering all its data members, member functions, conversion functions
400 and so on. However, the base classes won't be unregistered, since they don't
401 necessarily depend on it. Similarly, implicitly generated types (as an example,
402 the meta types implicitly generated for function parameters when needed) won't
403 be unregistered.
404 
405 To unregister a type, users can use the `unregister` function from the global
406 namespace:
407 
408 ```cpp
409 entt::unregister<my_type>();
410 ```
411 
412 This function returns a boolean value that is true if the type is actually
413 registered with the reflection system, false otherwise.<br/>
414 The type can be re-registered later with a completely different name and form.
